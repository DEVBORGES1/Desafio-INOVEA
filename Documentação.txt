Documentação do Desafio Técnico - primeiro passo
A documentação a seguir descreve o processo de solução do desafio técnico cujo foco é a construção de uma aplicação em que estou desenvolvendo para um desafio.

Pilha Tecnológica Exigida:

Banco de Dados: Microsoft SQL Server
Back-end: VB.NET (ASP.NET WebForms)
Front-end: JavaScript em conjunto com a biblioteca externa Telerik UI for ASP.NET AJAX
Fonte de Dados: Movies.csv (Kaggle)


Desafio 1: Estruturação do Banco de Dados e Importação
A primeira etapa do desafio consistiu inteiramente em arquitetar e popular as tabelas que irão sustentar a aplicação principal.


Obstáculo 1: Definição de Identificador Único
Desafio: Como garantir que cada filme pudesse ser rastreado sem ambiguidade pelo código da aplicação de forma padronizada.
Conhecimento Aplicado: Adoção do conceito de Chave Primária (Primary Key). Durante a modelagem visual, a coluna id do CSV foi designada no banco como PK, impedindo registros duplicados e servindo como "âncora" para todas as futuras buscas no sistema VB.NET.
Obstáculo 2: Ausência de Instância do Banco
Desafio: Ao inicializar a ferramenta SSMS (SQL Server Management Studio), obteve-se o erro: "O servidor não foi encontrado ou não estava acessível (Provider: Named Pipes Provider, error: 40)".
Causa e Resolução: Foi diagnosticado que o computador possuía apenas o programa de gerenciamento (a interface visual), mas faltava o "Motor" do banco. A situação foi superada instalando-se a instância do SQL Server Express. A partir disso, a autenticação foi estabelecida via Windows Authentication no diretório .\SQLEXPRESS.
Obstáculo 3: Higienização e Tipagem de Dados no Cadastro (DBNull)
Desafio: O assistente de importação de banco de dados travava sistematicamente durante a inserção com a mensagem: "A coluna 'overview' não permite DBNull.Value".
Causa e Resolução:
A importação vinha falhando ao encontrar filmes do Movies.csv em que a equipe do Kaggle não preencheu o resumo da história na coluna "overview". A regra foi alterada para a liberação de valores nulos, marcando-se a opção Allow Nulls (Permitir Nulos).
Ajuste do Data Type: Como a referida coluna contemplava descrições gigantescas, o tamanho do campo em banco foi alterado preventivamente para nvarchar(MAX), evitando o corte prematuro de dados textuais e falhas por limite de inserção.

Status Atual: A fase de infraestrutura da informação foi finalizada 100% com sucesso. O ambiente local está em conformidade e preparado para ser integrado à aplicação VB.NET.

------------------------------------------------------

Desafio 2: Renderização de Dados no Front-end com Telerik RadGrid
A segunda etapa exigiu conectar o banco de dados à interface do usuário via código (VB.NET), exibindo os dados em um grid avançado do Telerik.

Obstáculo 1: Configuração e Alvo da Instância do Banco
Desafio: Ligar o "Data Source" do Web.config à mesma instância do banco instalada anteriormente. Inicialmente o sistema acusou o "Erro 26 (Instância não localizada)", indicando que '.\SQLEXPRESS' estava incorreto.
Causa e Resolução: A instância do banco instalada foi a "Instância Padrão" (MSSQLSERVER) e não a versão nomeada. A linha de conexão no Web.config (connectionStrings) foi corrigida utilizando o '.' (ponto) no Data Source, que significa "servidor local principal".

Obstáculo 2: O erro 403.14 Forbidden no carregamento inicial
Desafio: O Visual Studio e IIS Express recusaram a exibição da tela inicial, gerando um erro de listagem de diretório negada.
Causa e Resolução: A ferramenta estava tentando abrir a "pasta raiz" do projeto ao invés da página alvo. A solução técnica foi acessar diretamente a URL completa finalizando com '/Projeto.aspx', garantindo que o servidor servisse o arquivo de destino.

Obstáculo 3: Carregamento Performático do Telerik (Advanced Data Binding)
Desafio: A tag antiga SqlDataSource foi removida e o RadGrid precisava receber os dados pesados sem travar a tela.
Conhecimento Aplicado: Adotamos o padrão recomendado "Advanced Data Binding". No Front-end (.aspx), informamos o evento 'OnNeedDataSource' dentro da tag do grid. No Backend (VB.NET), importamos a biblioteca 'System.Data.SqlClient' e escrevemos o comando SQL 'SELECT', jogando o resultado dentro de um 'DataTable' e setando o 'GridFilmes.DataSource'.

Obstáculo 4: Erro do Telerik (WebResource.axd is missing)
Desafio: A página abriu, porém estourou logo de cara exigindo um HttpHandler no Web.config.
Causa e Resolução: Componentes modernos (como o RadScriptManager) precisam de permissões especiais de injeção de Javascript no navegador para funcionarem. Adicionamos a tag `<system.webServer>` no final do Web.config liberando o WebResource especial da Telerik.

Status Atual: Módulo 2 finalizado 100% com sucesso. Banco de dados e Front-end interligados, exibindo lindamente o Grid populado com as informações processadas pelo ASP.NET.

Desafio 3: Internacionalização / Tradução (L10N) do RadGrid
O terceiro passo consistiu em traduzir os componentes interativos do RadGrid (Paginador, Tooltips de Filtro e Ordenação) da sua língua nativa (inglês) para o Português (pt-BR).

Obstáculo 1: Configuração do Dicionário Global (Global Resources)
Desafio: A plataforma ASP.NET e o Telerik por padrão não traduzem strings de filtro e ordenação sem os arquivos XML de recursos (.resx) apropriados.
Conhecimento Aplicado: Explorou-se o conceito nativo da Microsoft (A pasta 'App_GlobalResources'). Foi criado um dicionário 'RadGrid.Main.pt-BR.resx' no formato Tabela Visual (Resource Editor do Visual Studio) alimentado por pares de Chave/Valor oficiais da Telerik, como 'FilterEqualTo' -> 'Igual a'.

Obstáculo 2: Cache de Compilação e Arquivos Neutros (Gotcha do IIS)
Desafio: Apesar do arquivo '.pt-BR' e do código 'Culture="pt-BR"' estarem configurados, a aplicação insistia em não carregar as traduções e manter o inglês (Fallback).
Causa e Resolução: Em projetos Web Site clássicos do compilador ASP.NET, a classe 'ResourceManager' aborta a checagem caso o arquivo base / neutro correspondente não exista no diretório. A falha do sistema foi solucionada duplicando o arquivo e o renomeando para 'RadGrid.Main.resx' (neutro), obrigando o IIS a destravar o serviço de leitura após um 'Rebuild'. 

Obstáculo 3: Interceptação e Tradução Manual via Code-Behind (Solução Definitiva)
Desafio: Em vez de depender das inconstâncias do dicionário XML e do cache do servidor IIS, foi exigido uma rota de programação direta para forçar a interface independente da máquina.
Conhecimento Aplicado: No Front-end (.aspx), configuraram-se manualmente as propriedades filhas 'PagerStyle' e 'SortingSettings'. Já em relação ao Menu de Filtros, interceptou-se o evento nativo OnInit ('GridFilmes_Init') do RadGrid dentro do VB.NET. Através de um laço de repetição dinâmico (For Each), os campos subjacentes 'RadMenuItem' foram fisgados e renomeados programaticamente em tempo de execução. Esta intervenção por Back-end sobrepõe eventuais erros de cache na interface.

Obstáculo 4: Conflito de Codificação (Caracteres e Acentos Quebrados - "ContÃ©m")
Desafio: Ao inserir as palavras assentuadas direto pelo código C#/VB.NET, o compilador do servidor as transformou em caracteres ilegíveis durante a renderização (bug de UTF-8 vs ANSI).
Causa e Resolução: Isso ocorreu por uma assimetria na interpretação de formato de arquivo. O problema de codificação clássico foi atacado de frente forçando a leitura correta de UTF-8 direto no coração da aplicação (Web.config) utilizando a tag '<globalization fileEncoding="utf-8" />' além do salvamento nativo explícito da ferramenta IDE.

Status Atual: Desafio 3 concluído perfeitamente. O RadGrid opera maravilhosamente bem nos padrões de linguagem PT-BR, garantindo total conforto para usuários em português.

Desafio 4: Habilitação do CRUD (Create, Read, Update, Delete)
O quarto passo marcou a transformação da aplicação de uma mera tabela de leitura para um painel gerencial administrativo completo, permitindo a total manipulação de dados sobre o Telerik RadGrid.

Obstáculo 1: O Mapeamento Declarativo do Frontend e Configurações Mestras
Desafio: A propriedade genérica AutoGenerateColumns não permitia controle sobre o design da grade nem suportava a injeção nativa de botões de controle de idioma específico. 
Conhecimento Aplicado: No ambiente do 'Projeto.aspx', o Grid foi inteiramente remontado. A tag '<MasterTableView>' recebeu as essenciais definições 'CommandItemDisplay=Top' (habilitando a barra de ferramentas de inserção) e 'DataKeyNames=id', uma obrigação lógica do framework para reter a chave primária da linha que os comandos de edição atuarão. Foram geradas declarações precisas como '<telerik:GridEditCommandColumn>' para prover opções modulares em uma estrutura fixa controlada.

Obstáculo 2: A Interação ADO.NET Segura contra Hackers e a extração via Hashtable
Desafio: Transacionar as requisições enviadas ao Grid no Frontend de forma pura até o SQL local ('MoviesDB'), garantindo que ataques comuns, como Injeção de SQL (SQL Injection), fossem impossíveis.
Conhecimento Aplicado: Através do 'Projeto.aspx.vb', os três gatilhos morais ('OnUpdateCommand', 'OnInsertCommand', 'OnDeleteCommand') ativaram os scripts Backend. Para resgatar tudo o que foi modificado pelo usuário nas TextBoxes virtuais do grid, foi implementado o método Telerik intrínseco 'item.ExtractValues(Hashtable)'. Já na ligação de arquitetura ADO.NET com o Banco, implementaram-se comandos SQL nativos e atrelou-se a tecnologia de parâmetros (AddWithValue), que encapsula as strings do client em um contêiner fechado blindando todos os campos antes de acionar a 'SqlConnection'. Uma descoberta afortunada se mostrou: por o ID do SQL ser pré-configurado como 'Auto-Incremento', a coluna homônima foi intencionalmente delatada na exibição, deixando a inteligência de inserção dos valores sob inteira responsabilidade do T-SQL.

Status Atual: Desafio 4 finalizado com nota máxima. Os pilares de Criação, Leitura, Atualização e Destruição via ASP.NET Webforms operam perfeitamente. O projeto agora é uma aplicação de Dados Reais.

------------------------------------------------------

Desafio 5: Interações no Client-Side via JavaScript
Nesta etapa, focou-se em transferir e mesclar a responsabilidade dos eventos de resposta imediata do painel de administração para o navegador (Client-Side), aprimorando a dinamicidade da Web.

Obstáculo 1: A Captura de Dados Telerik Híbrida via JavaScript DOM
Desafio: Ativar a seleção unificada de células do Grid e interceptar um duplo clique veloz (OnRowDblClick), localizando qual filme foi acessado exclusivamente via navegador sem fazer "Reload" (chamadas ao C#).
Conhecimento Aplicado: No design do projeto via '<ClientSettings>', foi ativada a restrição '<Selecting AllowRowSelect="true" />', agrupando e totalizando a seleção de todas as células de uma linha da tabela de uma só vez. Estabelecido o escutador principal ('<ClientEvents OnRowDblClick="DuploCliqueJavascript" />'), a arquitetura Javascript montada sob um '<telerik:RadCodeBlock>' permitiu dominar os parâmetros soltos da interface Client. Com os comandos puros 'args.get_tableView().get_dataItems()', gerou-se um Array Mestre onde, com o rastreio numérico da execução atual ('args.get_itemIndexHierarchical()'), fisgamos virtualmente o componente inteiro manipulado pelo mouse. Segurando o próprio objeto do sistema, ordenou-se extrair diretamente a propriedade desejada ('get_cell("title").innerHTML'), processando uma formatação pura via Javascript (alert pop-up) sem qualquer erro de indexação. 

Status Atual: Desafio 5 finalizado. O RadGrid assumiu uma responsabilidade híbrida de ouro: O Backend atira pesadamente o banco SQL Server de forma oculta nos fundos (CRUD) enquanto o navegador abraça toda a beleza do Frontend flutuando com scripts e feedbacks dinâmicos.
